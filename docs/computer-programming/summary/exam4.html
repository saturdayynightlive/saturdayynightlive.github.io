Q1기초 아래 코드의 출력은?
int x = 7;
double y = x;          // (A) widening
int z = (int) 3.9;   // (B) narrowing
System.out.println(y + " " + z);
  • 7 3
  • 7.0 3
  • 7.0 4
  • 7 4
정답 보기
② 7.0 3
(A) int → double은 widening(자동 형변환)이라 7이 7.0으로 저장된다.
(B) double → int는 narrowing(명시적 캐스팅 필요)이며, 소수점 이하는 버림(반올림 아님)이라 3.9 → 3.
출력: 7.0 3.
Q2중간 아래 코드의 출력은?
String a = "hi";
String b = "hi";
String c = new String("hi");
System.out.print((a == b) + " ");
System.out.print((a == c) + " ");
System.out.print(a.equals(c));
  • true true true
  • true false true
  • false false true
  • true false false
정답 보기
② true false true
리터럴 "hi"는 String Pool에 한 번만 만들어져 a와 b가 같은 객체를 가리킨다 → a == b는 true.
new String("hi")는 힙에 새 객체를 만들므로 a와 주소가 다르다 → a == c는 false.
equals()는 내용을 비교하므로 a.equals(c)는 true.
== 은 참조(주소) 비교, equals는 내용 비교임을 구분할 것.

Q4중간 접근 제어자에 대한 설명으로 옳은 것은?
  • protected 멤버는 같은 패키지에서는 접근할 수 없고 자식 클래스에서만 접근할 수 있다
  • 아무 접근 제어자도 쓰지 않으면(default) 같은 패키지 안에서만 접근할 수 있다
  • private 멤버는 같은 패키지의 다른 클래스에서 직접 접근할 수 있다
  • public 멤버는 같은 클래스 안에서만 접근할 수 있다
정답 보기
② 옳음
접근 범위: private(같은 클래스) ⊂ default(같은 패키지) ⊂ protected(같은 패키지 + 다른 패키지의 자식) ⊂ public(어디서나).

① 틀림 — protected는 같은 패키지에서도 접근 가능하다(default 범위를 포함).
③ 틀림 — private은 오직 같은 클래스 안에서만 접근 가능하다.
④ 틀림 — public은 어디서나 접근 가능하다.
Q5재출제 아래 코드의 출력은? (static hiding vs instance overriding)
class Engine {
    static String spec() { return "base"; }
    String label() { return "engine"; }
}
class Turbo extends Engine {
    static String spec() { return "turbo"; }
    @Override
    String label() { return "boosted"; }
}

Engine e = new Turbo();
System.out.println(e.spec() + " " + e.label());
  • base engine
  • turbo boosted
  • base boosted
  • turbo engine
정답 보기
③ base boosted
e.spec() → static 메서드는 hiding(정적 바인딩): 참조 타입 Engine의 spec() → "base".
e.label() → 인스턴스 메서드는 overriding(동적 바인딩): 실제 객체 Turbo의 label() → "boosted".
static 메서드는 오버라이딩이 아니라 hiding이라 참조 타입으로 결정된다(Lec6 표).
Q6중간 오버로딩(overloading)에 대한 설명으로 옳은 것은?
int    f(int a) { return a; }
double f(int a) { return a; }   // (X)
int    f(double a) { return 0; }
  • (X)는 반환 타입이 다르므로 정상적인 오버로딩이다
  • (X)는 첫 번째 f(int)와 시그니처가 같아 컴파일 에러다
  • 매개변수 타입이 달라도 메서드 이름이 같으면 오버로딩이 불가능하다
  • f(double a)는 f(int a)와 이름이 같아 컴파일 에러다
정답 보기
② 옳음
오버로딩 시그니처는 메서드 이름 + 매개변수 목록으로 결정되며 반환 타입은 제외된다.
(X) double f(int a)는 int f(int a)와 이름·매개변수가 모두 같아(반환 타입만 다름) 중복 정의이므로 컴파일 에러다.

③ 틀림 — 매개변수 타입이 다르면(f(double)) 정상 오버로딩이다.
④ 틀림 — f(double a)는 매개변수 타입이 달라 f(int a)와 공존 가능하다.
Q7재출제 아래 Queue 코드의 출력은? (add/poll/peek)

  • xyzz

Q8함정 예외 분류에 대한 설명으로 옳은 것은?
  • IOException은 Unchecked Exception이라 try-catch나 throws 없이도 컴파일된다
  • NullPointerException은 Checked Exception이라 반드시 처리하거나 선언해야 한다
  • ArithmeticException과 ClassCastException은 Unchecked Exception이다
  • FileNotFoundException은 Unchecked Exception이다
정답 보기
③ 옳음
Unchecked(RuntimeException 계열, 처리 강제 안 됨): NullPointerException, ArithmeticException, ClassCastException, IllegalArgumentException, IndexOutOfBoundsException.
Checked(처리/선언 강제): IOException, FileNotFoundException, SQLException.

① 틀림 — IOException은 Checked라 처리 또는 throws 선언이 필수다.
② 틀림 — NullPointerException은 Unchecked다.
④ 틀림 — FileNotFoundException은 Checked다.
Q9중간 C++ 코드에 대한 설명으로 옳은 것은?
struct A { int v; };
class  B { int v; };

A a; a.v = 1;   // (1)
B b; b.v = 1;   // (2)
  • (1), (2) 모두 정상 컴파일된다
  • (1)은 정상, (2)는 에러 — struct는 멤버가 기본 public, class는 기본 private이기 때문
  • (1)은 에러, (2)는 정상 — class만 멤버 접근이 허용된다
  • struct와 class는 완전히 동일하므로 둘 다 정상이다
정답 보기
② 옳음
C++에서 struct의 멤버 기본 접근 지정자는 public, class의 기본은 private이다.
(1) A의 v는 public이라 외부에서 a.v = 1 접근 가능.
(2) B의 v는 private이라 외부에서 b.v 접근 시 컴파일 에러.
기본 접근 지정자가 다르다는 점이 struct와 class의 핵심 차이다.
Q10중간 아래 C++ static 멤버 코드에 대한 설명으로 옳은 것은?
class Ticket {
public:
    static int count;
    Ticket() { count++; }
};
int Ticket::count = 0;   // (외부 정의)

Ticket t1, t2, t3;
cout << Ticket::count;
  • 출력은 0이다 — static 멤버는 객체마다 따로 존재한다
  • 출력은 3이다 — static 멤버는 모든 객체가 공유한다
  • int Ticket::count = 0; 줄은 불필요하며 없어도 된다
  • static 멤버는 t1.count처럼 객체로만 접근할 수 있다
정답 보기
② 출력은 3
static 멤버는 클래스당 하나만 존재하며 모든 객체가 공유한다. 생성자가 3번 호출되어 count가 0→1→2→3이 된다.
static 멤버 변수는 클래스 외부에서 int Ticket::count = 0;으로 반드시 정의해야 한다(③ 틀림 — 없으면 링크 에러).
Ticket::count처럼 클래스명으로 접근하는 것이 일반적이다(④ 틀림).

Q18중간 아래 C++ 코드의 출력을 쓰시오. (복사 생성자 호출 시점)
class Node {
public:
    Node() {}
    Node(const Node&) { cout << "copy "; }
};
void byRef(Node& n) {}
void byVal(Node n) {}

Node a;

byVal(a);     // (2)

Q19재출제 ★Lab 아래 제네릭 메서드는 컴파일 에러가 난다. <T extends Comparable<T>> 같은 bounded type parameter가 왜 필요한지 설명하고, 이것이 와일드카드 <?>와 어떻게 다른지 서술하시오. (Lab6)
static <T> T smaller(T a, T b) {
    return a.compareTo(b) < 0 ? a : b;  // 컴파일 에러
}
정답 보기
왜 에러인가: 제약 없는 T를 컴파일러는 Object로 취급한다. Object에는 compareTo()가 없어 호출이 컴파일 에러다.

bounded type parameter: <T extends Comparable<T>>로 선언하면 T가 Comparable을 구현했음이 보장되어 compareTo()를 안전하게 호출할 수 있다. 즉 타입 변수 T 자체에 상한(upper bound)을 거는 문법이다.
static <T extends Comparable<T>> T smaller(T a, T b) {
    return a.compareTo(b) < 0 ? a : b;
}
와일드카드와의 차이: <? extends T>는 메서드 파라미터 타입(예: List<? extends Number>)에 쓰는 "알 수 없지만 T의 하위 타입" 표현이고, 반환·재사용할 이름이 없다. 반면 <T extends X>는 타입 변수 T를 선언해 메서드 전체에서 그 타입을 이름으로 재사용(인자·반환에 동일 T)할 수 있다는 점이 다르다.
Q20재출제 아래 코드의 출력을 예측하고, static 메서드에서 인스턴스 변수에 접근하면 왜 컴파일 에러가 나는지 서술하시오.
class Meter {
    int reading = 5;        // 인스턴스 변수
    static int calls = 0;   // static 변수

    static void report() {
        calls++;
        System.out.println(reading);  // 문제 줄
    }
}
정답 보기
컴파일 에러 — 출력 없음. System.out.println(reading) 줄에서 컴파일이 실패한다.

이유: static 메서드는 특정 객체에 속하지 않고 클래스에 속한다. 호출 시점에 어떤 인스턴스의 reading인지 정할 수 없으므로(인스턴스 변수는 객체마다 따로 존재) 접근이 금지된다. 반면 calls는 static이라 클래스당 하나뿐이므로 접근 가능하다.

해결: 인스턴스 변수를 쓰려면 메서드를 인스턴스 메서드로 바꾸거나, static 메서드가 객체를 매개변수로 받아 m.reading처럼 접근해야 한다. "static에서는 static만 직접 접근 가능"이 원칙이다.
Q21재출제 아래 커스텀 예외 코드는 의도와 다르게 동작한다. 무엇이 잘못되었는지 지적하고, 예외 메시지를 getMessage()로 받을 수 있게 올바르게 수정하시오. (super(message))
class InvalidAgeException extends Exception {
    InvalidAgeException(String msg) {
        System.out.println(msg);   // 메시지를 그냥 출력만 함
    }
}
정답 보기
문제점: 생성자에서 메시지를 System.out.println으로 출력만 하고 부모(Exception)에게 전달하지 않았다. 그 결과 예외 객체의 메시지 필드가 비어 있어, 호출자가 catch (InvalidAgeException e)에서 e.getMessage()를 부르면 null이 나온다.

수정 — super(msg)로 메시지를 부모에 전달: 
class InvalidAgeException extends Exception {
    InvalidAgeException(String msg) {
        super(msg);   // 부모 Exception에 메시지 전달

}
이렇게 하면 getMessage()가 전달한 메시지를 반환한다. 커스텀 예외는 extends Exception(Checked) 또는 extends RuntimeException(Unchecked)로 만들고, 생성자에서 println이 아니라 super(message)로 메시지를 넘기는 것이 정석이다.
Q22중간 아래 코드에서 catch 블록의 순서가 잘못되어 컴파일 에러가 난다. 이유를 설명하고 올바른 순서로 고치시오.
try {
    process();
} catch (Exception e) {
    System.out.println("general");
} catch (IllegalArgumentException e) {   // 도달 불가
    System.out.println("arg");
}
정답 보기
이유: catch는 위에서 아래로 순서대로 검사하는데, 첫 catch가 상위 타입 Exception이라 모든 예외를 먼저 잡아 버린다. 그러면 그 아래 IllegalArgumentException catch는 절대 도달할 수 없는(unreachable) 코드가 되어 컴파일 에러다.

수정 — 더 구체적인(하위) 타입을 먼저: 
try {
    process();
} catch (IllegalArgumentException e) {  // 구체적 타입 먼저
    System.out.println("arg");
} catch (Exception e) {                 // 일반 타입 나중에
    System.out.println("general");
}
예외 catch는 항상 구체적인 하위 타입 → 일반적인 상위 타입 순서로 작성해야 한다.
Q23중간 아래 코드는 캡슐화를 지키지 못하고 있다. getter/setter와 private을 사용해 어떻게 개선할지 서술하고, 캡슐화의 이점을 설명하시오.
class Account {
    public int balance;   // 외부에서 직접 수정 가능
}

Account ac = new Account();
ac.balance = -5000;   // 음수 잔액도 막을 수 없음
정답 보기
문제: balance가 public이라 외부에서 아무 값(음수 등)이나 직접 대입할 수 있어 잘못된 상태를 막을 수 없다.

개선: 필드를 private으로 숨기고 getter/setter로 접근을 통제한다. 
class Account {
    private int balance;

    public void setBalance(int b) {

}
캡슐화 이점: ① setter에서 유효성 검사를 넣어 잘못된 값(음수)을 차단한다. ② 내부 표현을 바꿔도 외부 코드에 영향이 없다(정보 은닉). ③ 읽기 전용(getter만 제공) 등 접근 수준을 세밀하게 제어할 수 있다.
Q24응용 아래 C++ 클래스는 얕은 복사(shallow copy) 문제를 가진다. 무엇이 위험한지 설명하고, 깊은 복사(deep copy)를 하는 복사 생성자를 어떻게 작성해야 하는지 서술하시오.
class Buffer {
public:
    int* data;
    int size;
    Buffer(int n) : size(n) { data = new int[n]; }
    ~Buffer() { delete[] data; }
    // 복사 생성자 없음 → 컴파일러가 얕은 복사 제공
};
정답 보기
얕은 복사의 위험: 컴파일러가 만든 기본 복사 생성자는 data 포인터 값(주소)만 복사한다. 그러면 원본과 복사본이 같은 배열을 가리키게 되어, 한쪽이 소멸하며 delete[] data를 하면 다른 쪽은 이미 해제된 메모리를 가리킨다. 결국 이중 해제(double free)나 dangling pointer로 크래시가 난다.

깊은 복사 복사 생성자: 새 배열을 별도로 할당하고 원소를 하나씩 복사해 독립적인 메모리를 갖게 한다. 
Buffer(const Buffer& o) : size(o.size) {
    data = new int[size];           // 새 배열 할당
    for (int i = 0; i < size; i++)
        data[i] = o.data[i];        // 원소 값 복사
}
포인터 멤버를 가진 클래스는 얕은 복사를 피하기 위해 깊은 복사 복사 생성자를 직접 정의해야 한다.
