Design Patterns Lecture 12

Clean Code 6원칙 + SOLID 5원칙

왜 Clean Code인가?

개발자는 코드를 작성하는 시간보다 읽는 시간이 10배 더 많다.
동작하는 코드 ≠ 좋은 코드. Clean code는 생산성, 유지보수성, 신뢰성을 높인다.

Clean Code 6원칙

1. 명확한 이름 사용 (Clear Names)

변수·메서드·클래스 이름은 의도를 명확히 드러내야 한다.
· 변수/메서드: camelCase / 클래스: PascalCase
· 과도한 축약 금지, 잘못된 정보 전달 금지 
// Bad
int d;  // 경과 시간? 날짜?

// Good
int elapsedTimeInDays;

2. 개념당 단어 하나 (One Word per Concept)

같은 개념에는 일관된 단어를 사용. get/fetch/retrieve를 섞지 말 것.

3. 함수를 작게 (Small Functions)

함수는 한 가지 일만 해야 한다. 15줄 이상이면 분리를 고려.

4. 부작용 최소화 (Minimize Side Effects)

Side Effect: 함수가 자신의 범위 밖의 변수(static/인스턴스 변수)를 변경하는 것.
공유 상태 변경은 신중하게 다루어야 한다.

5. DRY 원칙 (Don't Repeat Yourself)

DRY: 코드를 반복하지 말 것. 반복된 코드는 수정 시 누락으로 버그가 생긴다. 
// Bad: 3개의 별도 print 함수
static void printInt(Integer i)  { System.out.println("Type: Integer, Value: " + i); }
static void printStr(String s)   { System.out.println("Type: String, Value: "  + s); }

// Good: 제네릭 하나로 통합
static void print(Object obj) {
    System.out.printf("Type: %s, Value: %s\n", obj.getClass().getSimpleName(), obj);
}

6. 명확한 주석 (Clear Comments)

주석을 쓰기 전에 먼저 코드 자체를 명확하게 만들 수 없는지 고민.
WHAT(무엇)이 아닌 WHY(왜)를 설명하는 주석만 가치 있다.

SOLID 5원칙

S — Single Responsibility Principle (SRP)

단일 책임 원칙: 클래스는 변경되어야 할 이유가 하나뿐이어야 한다.
하나의 클래스 = 하나의 책임(역할). 
// Bad: 한 클래스가 설정 변경 + 보안 검사 두 가지 역할
class UserSettingService {
    void changeEmail(User u) { if (checkAccess(u)) { ... } }
    boolean checkAccess(User u) { ... }
}

// Good: 책임 분리
class UserSettingService { void changeEmail(User u) { ... } }
class SecurityService     { boolean checkAccess(User u) { ... } }

O — Open-Closed Principle (OCP)

개방-폐쇄 원칙: 클래스는 확장에 열려 있고, 수정에 닫혀 있어야 한다.
새 기능 추가 시 기존 코드를 수정하지 않고 확장으로만 해결. 
// Bad: 새 도형 추가 시 AreaCalculator 수정 필요
// Good: Shape 인터페이스 구현으로 확장
interface Shape { double getArea(); }
class Rectangle implements Shape { public double getArea() { return w * h; } }
class Circle    implements Shape { public double getArea() { return r * r * Math.PI; } }
// 새 도형? → Shape를 implements하는 클래스만 추가하면 됨

L — Liskov Substitution Principle (LSP)

리스코프 치환 원칙: 서브클래스는 언제나 슈퍼클래스를 대체할 수 있어야 한다.
서브클래스가 슈퍼클래스의 기대를 위반하면 LSP 위반. 
// Bad: Penguin은 날 수 없는데 setAltitude()가 강제됨
class Penguin extends Bird {
    @Override
    public void setAltitude(double alt) { }  // 아무것도 안 함 → LSP 위반
}

// Good: FlyingBird 분리
abstract class Bird       { void draw(); }
abstract class FlyingBird extends Bird { void setAltitude(double alt); }
class Duck    extends FlyingBird { ... }
class Penguin extends Bird       { ... }  // setAltitude 없음

I — Interface Segregation Principle (ISP)

인터페이스 분리 원칙: 클래스는 사용하지 않는 인터페이스 메서드를 구현하도록 강제되어서는 안 된다.
큰 인터페이스 하나보다 작은 인터페이스 여러 개가 낫다. 
// Bad: UIList가 write/delete를 구현 강제
interface ArticleService { void list(); void write(); void delete(); }

// Good: 인터페이스 분리
interface ArticleListService  { void list(); }
interface ArticleWriteService { void write(); }
class UIList  implements ArticleListService  { public void list()  { } }
class UIWrite implements ArticleWriteService { public void write() { } }

D — Dependency Inversion Principle (DIP)

의존 역전 원칙: 구체적인 클래스가 아닌 추상(인터페이스)에 의존해야 한다.
고수준 모듈이 저수준 모듈에 직접 의존하면 안 됨. 
// Bad: Foo가 Logger 구현체에 직접 의존
class Foo { private Logger logger = new Logger(); }

// Good: ILogger 인터페이스에 의존
interface ILogger { void log(String msg); }
class Foo {
    private ILogger logger;
    void setLogger(ILogger impl) { this.logger = impl; }
}

SOLID 한눈에 요약

원칙핵심 한 줄
SRP클래스는 변경 이유가 하나여야 한다
OCP확장에 열리고, 수정에 닫혀야 한다
LSP서브클래스는 슈퍼클래스를 대체할 수 있어야 한다
ISP안 쓰는 인터페이스 메서드를 구현 강제하면 안 된다
DIP구현이 아닌 추상에 의존해야 한다
