Q1중간 아래 코드의 출력을 쓰시오.
try {
    System.out.println("1");
    try {
        System.out.println("2");
        throw new RuntimeException();
    } finally {
        System.out.println("3");
    }
} catch (RuntimeException e) {
    System.out.println("4");
} finally {
    System.out.println("5");
}
정답 보기
1
2
3
4
5
안쪽 try에서 예외 발생 → 안쪽 finally("3") 먼저 실행 → 예외가 바깥 catch로 전파 → catch("4") 실행 → 바깥 finally("5") 실행.
finally는 예외 발생 여부 관계없이 항상 실행되며, 중첩 try에서도 안쪽부터 순서대로 실행된다.
Q2재출제 아래 코드에서 컴파일 에러가 나는 줄은?
class Sensor {}
class TempSensor extends Sensor {}
class Router {}

Sensor s = new TempSensor();
boolean b1 = s instanceof TempSensor;   // (A)
boolean b2 = s instanceof Router;       // (B)
Router r = (Router) s;                  // (C)
  • (A)만 에러
  • (B), (C) 둘 다 컴파일 에러
  • (C)만 런타임 에러
  • 에러 없음
정답 보기
② (B), (C) 둘 다 컴파일 에러
Router는 Sensor와 상속 관계가 전혀 없다. 관계 없는 타입에 대한 instanceof(B)와 캐스팅(C)은 컴파일러가 "incompatible types"로 거른다(런타임이 아니라 컴파일 단계).
(A)는 TempSensor가 Sensor 하위 타입이라 정상이며 결과는 true.
Q3재출제 다음 중 제네릭에 대한 설명으로 옳은 것은?
  • 제네릭 클래스를 사용할 때 타입 파라미터를 생략(raw type)해도 컴파일 에러는 발생하지 않는다
  • <T extends Number>로 선언된 T는 Number의 메서드를 호출할 수 없다
  • 제네릭 메서드는 클래스 전체가 제네릭이어야만 정의할 수 있다
  • <? extends T>와 <T extends X>는 동일한 구문이다
정답 보기
① 옳음
raw type(타입 파라미터 생략)은 컴파일 에러가 아닌 경고(unchecked warning)만 발생한다. 하지만 타입 안전성이 깨져 런타임에 ClassCastException이 발생할 수 있다.

② 틀림 — <T extends Number>는 T가 Number의 하위 타입임을 보장하므로 Number의 메서드(doubleValue() 등)를 호출할 수 있다.
③ 틀림 — 제네릭 메서드는 일반 클래스 안에도 정의할 수 있다(public static <E> void print(E[] arr)).
④ 틀림 — <? extends T>는 와일드카드(메서드 파라미터용), <T extends X>는 타입 변수 자체에 상한을 두는 bounded type parameter로 용도가 다르다.
Q4중간 C++ 초기화 리스트에 대한 설명으로 옳은 것은?
class Motor {
public:
    int rpm;
    int volt;
    Motor(int r, int v) : rpm(r), volt(v) {}
};
  • 초기화 리스트는 생성자 본문보다 나중에 실행된다
  • 초기화 리스트와 본문 대입이 모두 있으면 본문 대입이 우선된다
  • 초기화 리스트는 생성자 본문보다 먼저 실행되며, 둘 다 있으면 초기화 리스트가 우선된다
  • 초기화 리스트는 const 멤버 변수 초기화에 사용할 수 없다
정답 보기
③ 옳음
초기화 리스트(: rpm(r), volt(v))는 생성자 본문 실행 전에 처리된다.
같은 멤버에 초기화 리스트와 본문 대입이 모두 있으면 초기화 리스트가 우선된다.

① 틀림 — 본문보다 먼저 실행된다.
② 틀림 — 초기화 리스트가 우선이다.
④ 틀림 — const 멤버와 레퍼런스 멤버는 초기화 리스트로만 초기화할 수 있다(본문에서는 대입 불가).
Q5재출제 아래 코드의 출력은?
class Device {
    static String type() { return "Device"; }
    String name() { return "Generic"; }
}
class Phone extends Device {
    static String type() { return "Phone"; }
    @Override
    String name() { return "iPhone"; }
}

Device d = new Phone();
System.out.println(d.type() + " " + d.name());
  • Device Generic
  • Phone iPhone
  • Device iPhone
  • Phone Generic
정답 보기
③ Device iPhone
d.type() → static 메서드는 정적 바인딩(hiding): 참조 타입 Device의 type() → "Device".
d.name() → 인스턴스 메서드는 동적 바인딩(overriding): 실제 객체 Phone의 name() → "iPhone".
static 메서드는 오버라이딩이 아니라 hiding — 참조 타입으로 결정된다.
Q6중간 PriorityQueue에 대한 설명으로 옳은 것은?
  • 삽입한 순서(FIFO) 그대로 꺼내진다
  • 기본 정렬은 자연 순서이며, poll()은 우선순위가 가장 높은(기본은 최솟값) 원소를 꺼낸다
  • 생성자에 Comparator를 줄 수 없다
  • peek()은 가장 마지막에 들어온 원소를 반환한다
정답 보기
②
PriorityQueue는 힙 기반으로, 삽입 순서가 아니라 우선순위 순으로 꺼낸다. 기본은 자연 순서의 최솟값이 먼저 나온다.
생성자에 Comparator를 넘겨 정렬 기준을 바꿀 수 있다(예: 최대 힙은 Comparator.reverseOrder()). peek()은 다음에 나올(최우선) 원소를 제거 없이 반환.

Q8함정 아래 코드에 대한 설명으로 옳은 것은?
abstract class Printer {
    abstract void print(String doc);
    void banner() { System.out.println("=== PRINT ==="); }
}

abstract class NetworkPrinter extends Printer {
    abstract void connect(String ip);
    // print()는 구현 안 함
}

class LaserPrinter extends NetworkPrinter {
    void connect(String ip) { System.out.println("connected: " + ip); }
    void print(String doc) { System.out.println("printing: " + doc); }
}
  • NetworkPrinter가 print()를 구현하지 않았으므로 컴파일 에러다
  • LaserPrinter는 abstract가 아니므로 connect()만 구현하면 된다
  • LaserPrinter는 Printer의 print()와 NetworkPrinter의 connect() 두 추상 메서드를 모두 구현했으므로 정상 컴파일된다
  • abstract class는 생성자를 가질 수 없다
정답 보기
③ 옳음
추상 클래스는 추상 메서드를 구현하지 않고 다시 추상 클래스로 상속할 수 있다. NetworkPrinter는 abstract이므로 Printer의 print()를 넘겨도 된다.
구체 클래스(LaserPrinter)는 상속 체인에 남은 모든 추상 메서드를 구현해야 한다: Printer의 print() + NetworkPrinter의 connect(). 둘 다 구현했으므로 정상이다.

① 틀림 — NetworkPrinter 자체가 abstract이므로 미구현 추상 메서드를 그대로 가질 수 있다.
② 틀림 — LaserPrinter는 connect()뿐 아니라 부모 체인의 print()도 구현해야 한다.
④ 틀림 — abstract class는 생성자를 가질 수 있다(직접 new는 불가하지만 하위 클래스가 super()로 호출한다).
Q9중간 아래 C++ 코드의 출력은?
class Counter {
public:
    int val;
    Counter(int v) : val(v) {}
    Counter& add(int n) {
        val += n;
        return *this;
    }
    void show() { cout << val; }
};

Counter c(10);
c.add(3).add(2).show();
  • 10
  • 13
  • 15
  • 컴파일 에러
정답 보기
③ 15
add()는 *this를 레퍼런스로 반환하므로 메서드 체이닝이 가능하다.
c.add(3) → val = 13, 자기 자신 반환 → .add(2) → val = 15, 반환 → .show() → 15 출력.
C++에서 this는 현재 객체를 가리키는 포인터이며, *this는 그 객체 자체(레퍼런스 반환 시 사용).
Q10재출제 아래 C++ 코드에서 복사 생성자가 호출되는 횟수는?
class Img {

  • 2회

정답 보기

Q18중간 아래 C++ 코드의 출력을 쓰시오.

Q19재출제 아래 코드의 출력을 예측하고, super.describe() 호출이 없을 때와 비교하여 super 키워드의 역할을 서술하시오.
class Vehicle {
    String brand = "Generic";
    void describe() {
        System.out.println("Vehicle: " + brand);
    }
}
class Truck extends Vehicle {
    int payload;
    Truck(String brand, int payload) {
        this.brand = brand;
        this.payload = payload;
    }
    @Override
    void describe() {
        super.describe();
        System.out.println("Payload: " + payload + "t");
    }
}

new Truck("Volvo", 20).describe();
정답 보기
출력:
Vehicle: Volvo
Payload: 20t

super.describe() 역할: 오버라이드한 메서드 안에서 부모 클래스의 같은 이름 메서드를 명시적으로 호출한다. super.describe()가 없으면 "Vehicle: Volvo" 줄은 출력되지 않고 "Payload: 20t"만 출력된다.

super vs this: this.describe()로 쓰면 동적 바인딩으로 자기 자신(Truck의 describe)을 무한 재귀 호출하므로 반드시 super를 사용해야 한다. super는 부모의 구현을 재사용하면서 자식이 추가 동작을 덧붙이는 전형적인 상속 패턴이다.
Q20재출제 아래 코드가 컴파일되지 않는 이유를 설명하고, 두 가지 해결 방법을 제시하시오.
void loadConfig() {
    BufferedReader br =
        new BufferedReader(new FileReader("app.cfg"));
    String first = br.readLine();
    System.out.println(first);
}
정답 보기
이유: FileReader 생성자와 readLine()은 Checked Exception인 IOException(FileReader는 FileNotFoundException 포함)을 던질 수 있다. Checked Exception은 반드시 처리하거나 선언해야 하는데, 이 메서드는 둘 다 하지 않아 컴파일 에러가 난다.

Q22중간 아래 두 클래스를 보고, Comparable과 Comparator의 차이를 설명하고 각각 언제 사용하는지 서술하시오.
// (A)
class Book implements Comparable<Book> {
    String title; int pages;
    Book(String t, int p) { title = t; pages = p; }
    @Override
    public int compareTo(Book o) { return this.pages - o.pages; }
}

// (B)
Comparator<Book> byTitle =
    (a, b) -> a.title.compareTo(b.title);

List<Book> list = new ArrayList<>();
list.add(new Book("Java", 500));
list.add(new Book("Algo", 300));
Collections.sort(list);           // (A) 사용
list.sort(byTitle);               // (B) 사용
정답 보기
Comparable (A): 클래스 자신에 기본(자연) 정렬 기준을 내장한다. compareTo()를 구현하면 Collections.sort()나 TreeSet 등이 별도 Comparator 없이 정렬할 수 있다. 클래스 소스를 수정할 수 있고, 정렬 기준이 하나로 고정될 때 적합하다.

Comparator (B): 클래스 외부에서 다양한 정렬 기준을 별도로 정의한다. 클래스를 수정하지 않고도 여러 기준(제목순, 페이지순 등)으로 정렬할 수 있다. 기준이 여러 개이거나 클래스 소스를 수정할 수 없을 때 사용한다.

요약: Comparable은 클래스 내부("나는 이렇게 비교된다"), Comparator는 클래스 외부("이 기준으로 비교해라"). compareTo/compare 모두 음수·0·양수를 반환한다.
Q23중간 아래 추상 클래스는 일부 메서드만 구현하고 나머지는 추상으로 남겨 두었다. 이런 "부분 구현(템플릿)" 설계가 주는 이점을 서술하고, 이 클래스를 직접 new로 생성할 수 없는 이유를 설명하시오.
abstract class ReportJob {
    final void run() {        // 공통 흐름 고정
        load();
        String r = format();
        System.out.println(r);
    }
    void load() { System.out.println("loading"); } // 기본 구현
    abstract String format();   // 하위가 반드시 구현
}
정답 보기
이점(템플릿 메서드 패턴): 공통 처리 흐름(run()의 load → format → 출력 순서)은 상위에서 한 번만 정의해 고정하고, 달라지는 부분(format())만 하위 클래스가 채우게 한다. 흐름 중복을 없애고, 새 리포트 타입은 format()만 구현하면 되므로 확장이 쉽다.

new 불가 이유: 추상 메서드 format()은 본문이 없어 구현이 미완성이다. 미완성 클래스의 인스턴스를 만들면 format() 호출 시 실행할 코드가 없으므로, Java는 추상 클래스의 직접 인스턴스화를 컴파일 단계에서 금지한다. 모든 추상 메서드를 구현한 구체 하위 클래스만 생성할 수 있다.

요구사항: 재고 수량이 0 이하이면 OutOfStockException(커스텀 Checked Exception)을 던지는 reserve(int qty) 메서드. 호출자는 try-catch로 처리. 예외 메시지에는 요청 수량을 포함할 것.
정답 보기 
class OutOfStockException extends Exception {
    OutOfStockException(String msg) {
        super(msg);
    }
}

class Warehouse {
    int stock = 0;

    void reserve(int qty) throws OutOfStockException {
        if (stock <= 0) {
            throw new OutOfStockException("재고 없음: 요청 수량 " + qty);
        }
        stock -= qty;
    }
}

// 호출 예시
Warehouse wh = new Warehouse();
try {
    wh.reserve(3);
} catch (OutOfStockException e) {
    System.out.println(e.getMessage());
}
핵심 포인트:
① OutOfStockException extends Exception → Checked Exception이므로 reserve()에 throws OutOfStockException 선언 필수.
② 조건에 따라 throw new OutOfStockException(...)으로 실제 예외 객체를 생성·던진다.
③ 호출자는 catch (OutOfStockException e)로 반드시 처리해야 한다(처리 안 하면 컴파일 에러).
④ RuntimeException을 상속하면 Unchecked가 되어 throws 선언 없이도 쓸 수 있다.
Q27응용 아래 제네릭 메서드는 컴파일 에러가 난다. 무엇이 문제이고 어떻게 고쳐야 하는지(bounded type parameter) 코드로 보이시오.
// 리스트에서 최댓값을 찾으려는 의도
static <T> T max(List<T> list) {
    T best = list.get(0);
    for (T x : list)
        if (x.compareTo(best) > 0)   // 컴파일 에러
            best = x;
    return best;
}
정답 보기
문제: 타입 파라미터 T는 아무 제약이 없어 컴파일러가 T를 Object로 본다. Object에는 compareTo()가 없으므로 호출이 컴파일 에러다.

수정 — 상한 경계(bounded type parameter): 
static <T extends Comparable<T>> T max(List<T> list) {
    T best = list.get(0);
    for (T x : list)
        if (x.compareTo(best) > 0)
            best = x;
    return best;
}
<T extends Comparable<T>>로 T가 Comparable을 구현했음을 보장하면 compareTo() 호출이 가능해진다. extends는 클래스·인터페이스 모두에 쓴다.
