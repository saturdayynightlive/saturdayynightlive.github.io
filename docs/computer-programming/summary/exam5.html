Q1함정 아래 코드의 출력은?
class Vessel {
    String kind() { return "vessel"; }
    String tag()  { return kind() + "!"; }
}
class Tanker extends Vessel {
    @Override
    String kind() { return "tanker"; }
}

Vessel v = new Tanker();
System.out.println(v.tag());
  • vessel!
  • tanker!
  • vessel
  • 컴파일 에러
정답 보기
② tanker!
tag()는 Vessel에만 정의되어 그대로 상속·실행되지만, 그 안에서 부른 kind()는 동적 바인딩이라 실제 객체 Tanker의 kind()가 호출된다.
따라서 tag() 내부 kind() → "tanker" → tanker!.
상위 메서드 안에서 호출한 메서드도 오버라이딩되면 자식 버전이 불린다는 점이 함정.
Q2함정 다운캐스팅과 instanceof에 대해 옳은 것은?
class Animalia {}
class Feline extends Animalia {}
class Canine extends Animalia {}

Animalia x = new Feline();
Canine c = (Canine) x;   // (★)
  • (★)는 컴파일 에러다 — Feline과 Canine은 형제라 캐스팅 문법 자체가 불가능하다
  • (★)는 컴파일은 되지만 실행 시 ClassCastException이 발생한다
  • (★)는 컴파일·실행 모두 정상이며 c는 Feline을 가리킨다
  • x instanceof Canine은 true다
정답 보기
② 실행 시 ClassCastException
x의 정적 타입은 Animalia이고 Canine은 그 하위 타입이라 다운캐스팅 문법은 컴파일을 통과한다(① 틀림).
그러나 실제 객체는 Feline이고 Feline은 Canine이 아니므로 런타임에 ClassCastException이 터진다(③ 틀림).
x instanceof Canine은 실제 객체가 Feline이라 false다(④ 틀림). instanceof로 먼저 검사 후 캐스팅하는 것이 안전하다.
Q3재출제 아래 코드의 출력은? (static hiding vs instance overriding)
class Relay {
    static String code() { return "R"; }
    String name() { return "relay"; }
}
class Switcher extends Relay {
    static String code() { return "S"; }
    @Override
    String name() { return "switcher"; }
}

Relay r = new Switcher();
Switcher s = new Switcher();
System.out.println(r.code() + r.name() + s.code());
  • Rswitcher S
  • Sswitcher S
  • Rrelay R
  • Sswitcher R
정답 보기
① Rswitcher S
r.code() → static 메서드는 hiding(정적 바인딩): 참조 타입 Relay → "R".
r.name() → 인스턴스 메서드는 overriding(동적 바인딩): 실제 객체 Switcher → "switcher".
s.code() → 참조 타입 Switcher → "S".
이어 붙이면 Rswitcher S. static은 참조 타입, 인스턴스는 실제 객체로 결정(Lec6 표).
Q4함정 아래 오버로딩 호출 h(b, b)의 결과로 옳은 것은?
class Base {}
class Sub extends Base {}

void h(Sub x, Base y)  { System.out.println("first"); }
void h(Base x, Sub y)  { System.out.println("second"); }

Sub b = new Sub();
h(b, b);
  • first
  • second
  • ambiguous — 컴파일 에러
  • 실행 시 예외 발생
정답 보기
③ ambiguous — 컴파일 에러
b는 Sub 타입이라 h(Sub, Base)와 h(Base, Sub) 두 메서드 모두에 똑같이 일치한다(Sub는 Base이기도 하므로).
어느 한쪽이 더 적합하다고 가릴 수 없어 컴파일러가 "reference to h is ambiguous" 에러를 낸다.
이렇게 두 오버로딩에 동시에 들어맞는 호출은 모호성(ambiguous) 컴파일 에러가 된다(Lec7).
Q5함정 Iterator와 ConcurrentModificationException에 대해 옳은 것은?
  • for-each 루프 도중 list.remove(x)를 호출해도 안전하다
  • Iterator의 remove()는 next()로 반환받은 직전 원소를 안전하게 제거한다
  • hasNext()는 다음 원소를 반환하면서 커서를 전진시킨다
  • ConcurrentModificationException은 Checked Exception이라 try-catch가 필수다
정답 보기
② 옳음
Iterator의 remove()는 바로 직전 next()가 돌려준 원소를 컬렉션에서 안전하게 제거한다.

① 틀림 — for-each(내부적으로 Iterator) 도중 컬렉션의 remove()를 직접 부르면 ConcurrentModificationException이 난다.
③ 틀림 — 커서를 전진시키며 원소를 반환하는 건 next()다. hasNext()는 다음 원소 존재 여부만 boolean으로 알려준다.
④ 틀림 — ConcurrentModificationException은 RuntimeException 계열(Unchecked)이다.
Q6함정 아래 메서드에 대해 컴파일 에러 없이 호출 가능한 것은?
// 상한 와일드카드: Number 또는 그 하위 타입의 리스트만 허용
static Double total(List<? extends Number> list) { ... }
  • total(new ArrayList<String>())
  • total(new ArrayList<Integer>())
  • total(new ArrayList<Object>())
  • total("123")
정답 보기
② total(new ArrayList<Integer>())
<? extends Number>는 "Number 또는 그 하위 타입"의 리스트만 받는다(상한 와일드카드, Lec7 표).
② Integer는 Number의 하위 타입 → 가능.
① String은 Number 하위가 아님 → 불가.
③ Object는 Number의 상위 타입이라 ? extends Number에 맞지 않음 → 불가.
④ 인자가 List가 아니라 String → 불가.
Q7함정 아래 C++ 코드에서 virtual 유무에 따른 출력으로 옳은 것은?
class Shape3 {
public:
    virtual void draw() { cout << "shape"; }
    void area() { cout << "S"; }   // virtual 아님
};
class Ring : public Shape3 {
public:
    void draw() override { cout << "ring"; }
    void area() { cout << "R"; }
};

Shape3* p = new Ring();
p->draw();
p->area();
  • ringR
  • shapeS
  • ringS
  • shapeR
정답 보기
③ ringS
draw()는 virtual이라 동적 바인딩 → 실제 객체 Ring의 draw() → "ring".
area()는 virtual이 아니라 정적 바인딩 → 포인터 타입 Shape3의 area() → "S".
출력 ringS. C++은 Java와 달리 virtual을 붙인 함수만 동적 바인딩되고, 안 붙이면 포인터 타입으로 결정된다.
Q8함정 아래 C++ 코드의 출력은? (복사 생성자 호출 횟수)
class Parcel {
public:
    Parcel() {}
    Parcel(const Parcel&) { cout << "C"; }
};
void send(Parcel p) {}      // 값 전달
void track(Parcel& p) {}    // 레퍼런스 전달

int main() {
    Parcel a;
    Parcel b = a;     // (1) 초기화
    send(a);          // (2) 값 전달
    track(a);         // (3) 레퍼런스 전달
    return 0;
}
  • C
  • CC
  • CCC
  • (출력 없음)
정답 보기
② CC
(1) Parcel b = a → 동일 타입 객체로 초기화 → 복사 생성자 1회 → "C".
(2) send(a) → 값 전달이라 인자를 복사 → 복사 생성자 1회 → "C".
(3) track(a) → 레퍼런스 전달이라 원본을 그대로 참조 → 복사 안 함.
따라서 CC. 복사 생성자는 초기화·값 전달·값 반환 시 호출되며, 레퍼런스/포인터 전달은 복사하지 않는다.

Q9A/B 아래 Comparator 코드 A와 B 중 올바르게 동작하는 것은?
// A
class AscComp implements Comparator<Integer> {
    @Override
    public int compare(Integer a, Integer b) { return a - b; }
}

// B
class AscComp2 implements Comparator<Integer> {
    @Override
    public boolean compare(Integer a, Integer b) { return a < b; }
}
  • A가 올바르다
  • B가 올바르다
정답 보기
A가 올바르다
Comparator의 compare는 반드시 반환 타입이 int이고 인자가 2개여야 한다. 음수→a가 앞, 0→같음, 양수→b가 앞.
A는 public int compare(...) + a - b(오름차순)로 시그니처가 정확하다.
B는 반환 타입이 boolean이라 Comparator 인터페이스의 compare를 구현하지 못해 컴파일 에러다. (@Override가 있어도 시그니처 불일치로 막힌다.)
Q10A/B 아래 Comparable 코드 A와 B 중 올바르게 동작하는 것은?
// A
class Coin implements Comparable<Coin> {
    int value;
    @Override
    public int compareTo(Coin a, Coin b) { return a.value - b.value; }
}

// B
class Coin2 implements Comparable<Coin2> {
    int value;
    @Override
    public int compareTo(Coin2 o) { return value - o.value; }
}
  • A가 올바르다
  • B가 올바르다
정답 보기
B가 올바르다
Comparable의 compareTo는 인자가 1개(자기 자신 this와 인자 o를 비교)이고 반환 타입은 int다.
B는 public int compareTo(Coin2 o)로 시그니처가 정확하다.
A는 인자를 2개 받아(compareTo(Coin a, Coin b)) Comparable의 메서드가 아니다. @Override가 막아 컴파일 에러. (인자 2개로 비교하는 건 Comparator의 compare다.)
Q11A/B 아래 커스텀 예외 코드 A와 B 중 getMessage()가 전달 메시지를 올바로 반환하는 것은?
// A
class RangeError extends RuntimeException {
    RangeError(String msg) { super(msg); }
}

// B
class RangeError2 extends RuntimeException {
    RangeError2(String msg) { System.out.println(msg); }
}
  • A가 올바르다
  • B가 올바르다
정답 보기
A가 올바르다
커스텀 예외는 생성자에서 super(msg)로 메시지를 부모(RuntimeException/Exception)에 전달해야 getMessage()가 그 메시지를 반환한다.
A는 super(msg)를 호출하므로 올바르다.
B는 메시지를 System.out.println으로 출력만 하고 부모에 넘기지 않아, getMessage()가 null을 반환한다. println이 아니라 super(message)가 정석.
Q12A/B 주장 A와 B 중 옳은 것은? (C++ 상속 생성/소멸 순서)
// 3단계 상속: Grand → Mid → Leaf, 각 생성자/소멸자가 자기 이름 출력
Leaf obj;   // 지역 객체 1개
  • 생성은 Grand→Mid→Leaf, 소멸은 Leaf→Mid→Grand 순서다
  • 생성은 Leaf→Mid→Grand, 소멸은 Grand→Mid→Leaf 순서다
정답 보기
A가 옳다
C++에서 객체를 만들 때 생성자는 부모 → 자식 순서로 호출된다: Grand → Mid → Leaf.
소멸자는 그 역순(자식 → 부모)으로 호출된다: Leaf → Mid → Grand.
즉 출력은 Grand Mid Leaf 생성 후, 객체 소멸 시 ~Leaf ~Mid ~Grand. 생성과 소멸은 항상 정반대 순서다.
Q13A/B 아래 제네릭 메서드 선언 A와 B 중 문법이 올바른 것은? (bounded type, ★Lab6)
// A
static <T extends Number> double sum(T a, T b) {
    return a.doubleValue() + b.doubleValue();
}

// B
static double <T extends Number> sum(T a, T b) {
    return a.doubleValue() + b.doubleValue();
}
  • A가 올바르다
  • B가 올바르다
정답 보기
A가 올바르다
제네릭 메서드는 타입 파라미터 선언 <T extends X>가 반환 타입 앞에 와야 한다. 순서는 제어자 → <타입파라미터> → 반환타입 → 메서드명.
A: static <T extends Number> double sum(...) — 올바른 순서.
B: static double <T extends Number> sum(...) — 반환 타입과 <T>의 위치가 뒤바뀌어 컴파일 에러. <T extends Number> 덕분에 doubleValue() 호출이 가능하다.
Q14A/B 아래 catch 순서 A와 B 중 컴파일되는 것은?
// A
try { f(); }
catch (RuntimeException e) { }
catch (Exception e) { }

// B
try { f(); }
catch (Exception e) { }
catch (RuntimeException e) { }
  • A가 컴파일된다
  • B가 컴파일된다
정답 보기
A가 컴파일된다
catch는 위에서 아래로 검사하므로 구체적인(하위) 타입을 먼저, 일반적인(상위) 타입을 나중에 써야 한다.
A: RuntimeException(하위) → Exception(상위) 순서라 올바르다.
B: 상위 타입 Exception을 먼저 써서 모든 예외를 가로채므로, 아래 RuntimeException catch가 도달 불가(unreachable)가 되어 컴파일 에러다.

Q15복합추적 아래 코드의 출력을 쓰시오. (상속+오버라이딩+다운캐스팅+예외)
class Gadget {
    String use() { return "G"; }
}
class Drone extends Gadget {
    @Override
    String use() { return "D"; }
    String fly() { return "fly"; }
}

Gadget g = new Drone();
System.out.print(g.use());
try {
    Drone d = (Drone) g;          // (1)
    System.out.print(d.fly());
    Gadget g2 = new Gadget();
    Drone d2 = (Drone) g2;        // (2)
    System.out.print(d2.fly());
} catch (ClassCastException e) {
    System.out.print("X");
}
정답 보기
DflyX
g.use() → 실제 객체 Drone → "D".
(1) g의 실제 객체가 Drone이므로 다운캐스팅 성공 → d.fly() → "fly".
(2) g2의 실제 객체는 Gadget이라 Drone으로 다운캐스팅하면 ClassCastException 발생 → catch로 "X".
출력 순서: D fly X → DflyX. (2)의 예외로 그 뒤 d2.fly()는 실행되지 않는다.
Q16복합추적 아래 코드의 출력을 쓰시오. (super 생성자 + 오버라이딩)
class Unit {
    Unit() { System.out.print("U"); show(); }
    void show() { System.out.print("u"); }
}
class Soldier extends Unit {
    int hp = 100;
    Soldier() { super(); System.out.print("S"); }
    @Override

}

new Soldier();
정답 보기
U0S
new Soldier() → super()로 Unit 생성자 먼저 실행 → "U" 출력 후 show() 호출.
show()는 오버라이딩되어 Soldier의 것이 불리는데, 이 시점은 Unit 생성자 실행 중이라 Soldier의 hp = 100 초기화가 아직 안 됨 → hp는 기본값 0 출력.
그 후 Soldier 생성자 본문 → "S".
출력: U0S. 부모 생성자에서 오버라이딩 메서드를 부르면 자식 필드가 아직 0/null임이 함정.
Q17중간 아래 코드의 출력을 쓰시오. (try-catch-finally)
static int calc() {
    try {
        int x = 10 / 0;     // ArithmeticException
        return 1;
    } catch (ArithmeticException e) {
        System.out.print("C");
        return 2;
    } finally {
        System.out.print("F");
    }
}
System.out.print(calc());
정답 보기
CF2
10 / 0에서 ArithmeticException 발생 → catch 진입 → "C" 출력, 반환값 2를 준비.
return 2가 실행되기 전에 finally는 항상 실행된다 → "F" 출력.
그 후 2가 반환되어 호출부에서 "2" 출력.
순서: C → F → 2 → CF2. finally는 return 직전에도 반드시 실행된다.
Q18중간 아래 Map 코드의 출력을 쓰시오. (TreeMap 정렬)
Map<String, Integer> m = new TreeMap<>();
m.put("banana", 2);

{apple=9, banana=2, cherry=3}

Q19복합추적 아래 C++ 코드의 출력을 쓰시오. (가상 소멸자 + 상속 소멸 순서)
class Frame {
public:
    Frame() { cout << "cF "; }
    virtual ~Frame() { cout << "dF "; }
};
class Panel : public Frame {
public:
    Panel() { cout << "cP "; }
    ~Panel() { cout << "dP "; }
};

int main() {
    Frame* p = new Panel();
    delete p;
    return 0;
}
정답 보기
cF cP dP dF
new Panel() → 생성자는 부모 → 자식: Frame() "cF " → Panel() "cP ".
~Frame()가 virtual(가상 소멸자)이므로 delete p 시 실제 객체 Panel의 소멸자부터 호출된다 → 소멸은 자식 → 부모: ~Panel() "dP " → ~Frame() "dF ".
출력: cF cP dP dF. 가상 소멸자가 없었다면 ~Frame만 불려 "dF "만 나오고 자원 누수가 났을 것이다.
Q20복합추적 아래 C++ 코드의 출력을 쓰시오. (deep copy 복사 생성자 + 소멸자)
class Pack {
public:
    int* p;
    Pack(int v) { p = new int(v); cout << "C" << *p << " "; }
    Pack(const Pack& o) { p = new int(*o.p); cout << "K" << *p << " "; }
    ~Pack() { cout << "D" << *p << " "; delete p; }
};

int main() {
    Pack a(7);
    Pack b = a;     // 복사 생성
    return 0;
}
정답 보기
C7 K7 D7 D7
Pack a(7) → 일반 생성자 → "C7 ".
Pack b = a → 복사 생성자(deep copy: 새 int 할당) → "K7 ".
main 종료 시 생성 역순으로 소멸: b부터 ~Pack() → "D7 ", 그다음 a → "D7 ".
출력: C7 K7 D7 D7. deep copy라 a와 b가 서로 다른 메모리를 가리켜 각자 delete p해도 이중 해제가 없다.

5문제 (주관식)

Q21재출제 C++에서 얕은 복사(shallow copy)와 깊은 복사(deep copy)의 차이를 설명하고, 포인터 멤버를 가진 클래스에서 얕은 복사가 왜 위험한지(이중 해제 등) 서술하시오. (Q24 보강)
정답 보기
얕은 복사: 컴파일러가 자동 생성하는 기본 복사 생성자는 멤버를 그대로(비트 단위) 복사한다. 포인터 멤버라면 가리키는 주소(값)만 복사되어 원본과 복사본이 같은 힙 메모리를 공유하게 된다.

깊은 복사: 복사 생성자를 직접 정의해 새 메모리를 할당하고 그 내용을 복사한다. 원본과 복사본이 서로 독립적인 메모리를 가진다.

얕은 복사의 위험: 두 객체가 같은 메모리를 가리키므로 ① 한쪽에서 값을 바꾸면 다른 쪽도 영향을 받고, ② 소멸자에서 각자 delete를 호출하면 같은 메모리를 두 번 해제(double free)하여 크래시가 나며, ③ 한쪽이 먼저 해제되면 다른 쪽은 댕글링 포인터(dangling pointer)가 된다.

결론: 포인터 멤버를 가진 클래스는 깊은 복사 복사 생성자를 직접 정의해야 안전하다.
Q22중간 Java의 오버로딩(overloading)과 오버라이딩(overriding)의 차이를, 시그니처 조건과 바인딩 시점(정적/동적) 측면에서 비교 설명하시오.
정답 보기
오버로딩(overloading): 같은 클래스 안에서 메서드 이름은 같되 매개변수 목록(타입/개수)이 다른 메서드를 여러 개 두는 것. 반환 타입은 시그니처에 포함되지 않으므로 반환 타입만 다른 것은 오버로딩이 아니다(컴파일 에러). 어떤 메서드를 부를지는 컴파일 시점(정적 바인딩)에 인자 타입으로 결정된다.

오버라이딩(overriding): 상속 관계에서 부모의 메서드를 자식이 같은 시그니처로 재정의하는 것. 시그니처(이름+매개변수)와 반환 타입이 같아야 한다(@Override 권장). 실제 호출은 실행 시점(동적 바인딩)에 객체의 실제 타입으로 결정된다.

핵심 대비: 오버로딩 = 같은 클래스·매개변수 다름·정적 바인딩 / 오버라이딩 = 상속·시그니처 동일·동적 바인딩.
Q23중간 Checked Exception과 Unchecked Exception의 차이를 설명하고, 각각에 해당하는 예외를 2개씩 들어 분류 기준을 서술하시오. 또한 throw와 throws의 차이도 함께 설명하시오.
정답 보기
Checked Exception: 컴파일러가 처리(try-catch) 또는 선언(throws)을 강제하는 예외. 처리하지 않으면 컴파일 에러다. 예: IOException, FileNotFoundException, SQLException.

Unchecked Exception: RuntimeException 계열로, 처리·선언을 강제하지 않는다(주로 프로그래밍 실수). 예: NullPointerException, ArithmeticException, ClassCastException, IllegalArgumentException, IndexOutOfBoundsException.

분류 기준: RuntimeException을 상속하면 Unchecked, 그 외 Exception 하위는 Checked다.

throw vs throws: throw는 실제로 예외 객체를 던지는 실행문(throw new X())이고, throws는 메서드 선언부에서 이 메서드가 어떤 예외를 던질 수 있다고 알리는 선언이다(void f() throws IOException).
Q24중간 ArrayList와 LinkedList의 내부 구조 차이를 설명하고, 어떤 상황에서 각각이 유리한지(임의 접근 vs 중간 삽입/삭제) 서술하시오.
정답 보기
ArrayList: 내부적으로 배열로 원소를 저장한다. 인덱스로 임의 접근(get(i))이 O(1)로 빠르다. 그러나 중간에 삽입/삭제하면 뒤 원소들을 한 칸씩 밀어야 해서 O(n)으로 느리다.

LinkedList: 각 원소가 다음(이전) 노드의 참조를 가진 연결 리스트다. 중간 삽입/삭제는 참조만 바꾸면 되어 빠르다(노드를 찾은 뒤에는 O(1)). 그러나 인덱스 접근은 처음부터 순회해야 해서 O(n)으로 느리다.

선택 기준: 조회(임의 접근)가 많으면 ArrayList, 중간 삽입/삭제가 잦으면 LinkedList가 유리하다. 둘 다 List 인터페이스를 구현해 같은 메서드로 다룰 수 있다.
Q25중간 Clean Code 6원칙 중 3가지를 골라 설명하고, SOLID 원칙 중 단일 책임 원칙(SRP)이 무엇인지 예를 들어 서술하시오.
정답 보기
Clean Code 원칙(예시 3가지):
① 의미 있는 이름 — 변수·함수 이름만 보고 역할을 알 수 있게 짓는다(d 대신 elapsedDays).

Q27응용 아래 요구사항을 만족하는 Java 코드를 작성하시오.
요구사항: 정수 idx가 0 미만이거나 배열 길이 이상이면 커스텀 Unchecked 예외 BadIndexException(extends RuntimeException)을 던지는 at(int[] arr, int idx) 메서드. 예외 메시지에 잘못된 idx 값을 포함. 호출자는 try-catch로 처리.
정답 보기 
class BadIndexException extends RuntimeException {
    BadIndexException(String msg) {
        super(msg);   // println 아니라 super로 메시지 전달
    }
}

static int at(int[] arr, int idx) {
    if (idx < 0 || idx >= arr.length) {
        throw new BadIndexException("잘못된 인덱스: " + idx);
    }
    return arr[idx];
}

// 호출 예시
int[] a = {10, 20, 30};
try {
    System.out.println(at(a, 5));
} catch (BadIndexException e) {
    System.out.println(e.getMessage());   // 잘못된 인덱스: 5
}
핵심:
① extends RuntimeException → Unchecked이므로 throws 선언이 강제되지 않는다.
② 생성자에서 super(msg)로 메시지 전달해야 getMessage()가 동작한다.
③ Unchecked라 try-catch는 선택이지만, 위처럼 잡아 처리할 수 있다.
Q28응용 ★Lab 아래 요구사항을 만족하는 Java 코드를 작성하시오.
요구사항: 두 인자 중 큰 값을 돌려주는 제네릭 메서드 larger를, <T extends Comparable<T>> bounded type parameter로 작성. 메서드 안에서 compareTo로 비교. (Lab6)
정답 보기 
static <T extends Comparable<T>> T larger(T a, T b) {
    return a.compareTo(b) >= 0 ? a : b;
}

// 사용
System.out.println(larger(3, 8));        // 8
System.out.println(larger("ab", "az"));  // az
핵심:
① 타입 파라미터 선언 <T extends Comparable<T>>가 반환 타입 T 앞에 온다(순서: static <T extends ...> 반환타입 메서드명).
② 제약이 있어 T가 Comparable임이 보장되므로 a.compareTo(b) 호출이 안전하다(제약 없으면 Object로 보여 컴파일 에러).
③ compareTo는 음수/0/양수를 반환하며, >= 0이면 a가 b 이상이라 a를 반환.
Q29응용 아래 요구사항을 만족하는 Java 코드를 작성하시오.
요구사항: 문자열을 FileWriter로 "out.txt"에 쓰고 닫는 메서드. 작업 성공/실패와 무관하게 finally에서 "done"을 출력. IOException은 try-catch로 처리.
정답 보기 
import java.io.FileWriter;
import java.io.IOException;

static void writeFile(String text) {
    FileWriter fw = null;
    try {
        fw = new FileWriter("out.txt");
        fw.write(text);
        fw.close();
    } catch (IOException e) {
        System.out.println("쓰기 실패: " + e.getMessage());
    } finally {
        System.out.println("done");   // 항상 실행
    }
}
핵심:
① FileWriter의 생성·write·close는 IOException(Checked)을 던지므로 try-catch 또는 throws가 필요하다.
② finally는 예외 발생 여부와 무관하게 항상 실행되므로 "done"이 반드시 출력된다.
③ write로 문자열을 파일에 기록하고 close로 자원을 닫는다.
Q30응용 ★Lab 아래 C++ 코드는 다형성과 메모리 해제가 의도대로 동작하지 않는다. 현재 출력을 예측하고, virtual·override·가상 소멸자를 적용한 올바른 코드로 수정하시오. (Lab11)
class Media {
public:
    void play() { cout << "media "; }   // virtual 아님
    ~Media() { cout << "~media "; }       // virtual 아님
};
class Video : public Media {
public:
    void play() { cout << "video "; }
    ~Video() { cout << "~video "; }
};

Media* m = new Video();
m->play();
delete m;
정답 보기
현재 출력: media ~media
① play()가 virtual이 아니라 정적 바인딩 → 포인터 타입 Media의 play() → "media ".
② 소멸자가 virtual이 아니라 delete m 시 Media 소멸자만 호출 → "~media " (Video 소멸자 미호출 → 자원 누수 위험).

수정 — 기반 클래스 메서드·소멸자에 virtual: 
class Media {
public:
    virtual void play() { cout << "media "; }
    virtual ~Media() { cout << "~media "; }   // 가상 소멸자
};
class Video : public Media {
public:
    void play() override { cout << "video "; }
    ~Video() override { cout << "~video "; }
};
// 수정 후 출력: video ~video ~media
기반 클래스 포인터로 파생 객체를 다룰 때는 가상 소멸자가 있어야 delete 시 파생 → 기반 순으로 소멸자가 올바르게 호출된다.
